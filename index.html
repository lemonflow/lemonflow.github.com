General

lemonflow is a framework for structured holisitic prototyping of animated data-driven user interfaces & visualizations. The framework suits as a reference implementation of certain novel concepts. Exposing and applying such concepts in a easy and natural way is very helpful when creating holistic prototypes of the entire interactive experiences - i.e. going beyond single visualizations, components or screens. These structure and concepts serve as a guidance with in projects with changing ideas and quick reaction times.

lemonflow enables the developer to bind data sources to drive visual properties of the elements on screen. It is thereby inspired and based on the work of Jeffrey Heer and his visualization frameworks prefuse, flare, protovis and D3. lemonflow implements concepts from these frameworks and extends the framework in several aspects:

  • merge and intersect componentized UIs with free-flow visualizations
  • adds states & transisitons, supports explicit animations (tweener-like)
  • supports dynamic data collections and data streams
  • build-in prediction/recommendation engine
  • context-aware information processing

Versions & Source Language

Three versions of lemonflow exist, mainly contributing the same functionionality but for different languages/runtimes:

  • Java (depricated):
    compiles for and runs with adobe flash runtime, AVM2
  • JavaScript-QT (experimental):
    runs with javascript runtime provided as integrated engine for QML Runtime (QT version 5.0.1)
  • ActionScript:
    compiles for and runs with adobe flash runtime, AVM2
  • JavaScript:
    runs with javascript runtime for ECMAScript 5 - e.g. v8; the JS source files in this repo are automatically generated from TypeScript; TypeScript is a superset of JavaScript which roughly goes in line with ECMAScript 6

Visual Operators (Custom Layouts)


package lemonflow.components.menubar
{
  import lemonflow.ui.base.InteractiveOperator;
  import lemonflow.ui.base.VisualItem;
  
  public class OperatorBarLayout extends lemonflow.ui.base.InteractiveOperator
  {
    protected var _view:* = null;
    private var floatingIdx:Number = 0;
    
    public var itemSpacing:int = 87;
    public var totalWidth:int = 940;
    public var offsetLeft:int = 40;
    public var offsetRight:int = 40;
    
    public var itemsWidths:Array = new Array();
    public var itemsWidthSum:Number = 0;
    public var itemsPositions:Array = new Array();
    
    public function OperatorBarLayout(v:*) {
      _view = v;
    }
    
    override public function update(slots:Vector. = null, currentIdx:Number = 0, nextIndex:Number = 0):void {
      itemsWidthSum = 0;
      for(var i:int=0; i<_view.slots.length; i++) {
        itemsWidths[i] = MenuBarItemRenderer(_view.renderers[i]).textMask.width;
        itemsWidthSum+=itemsWidths[i];
      }
      
      itemSpacing = _view.dataProvider.length>0?((totalWidth-itemsWidthSum)-(offsetLeft + offsetRight))/(_view.dataProvider.length-1):0;
      
      for(var k:int=0; k<_view.slots.length; k++) {
        itemsPositions[k] = (k>0?(itemsWidths[k-1]+itemSpacing):0);
        if(k>0) itemsPositions[k] += itemsPositions[k-1];
      }
      
      for(var j:int=0; j<_view.slots.length; j++) {
        floatingIdx = j-_view.currentIndex;
        _view.slots[j].x =  offsetLeft+itemsPositions[j];
        _view.slots[j].y =  _view.centerXYZ.y + 4;
        _view.slots[j].visible = (floatingIdx<=5);
      }

    }
  }
}

UI Components vs. Visualizations

One aspect which lemonflow addreses is the artifical distinction between component-based user interfaces with a certain information architecture and flow (lists, menu bars, screens, tabs, side bars etc.) on one hand and then on the other hand dynamic data visualizations (e.g. a force-based layout of nodes). It might not be immediatly occurant why this is similar, but deep down this is just an artifical distinction which can be painful for developers. In many concrete modern user interfaces - the distinction between both types of visual information displays blurs and is often matter of discussion in concept phases. Authors want to use information visualizations as interactive components possible as their main metaphor of the entire UI. lemonflow suits both needs and integrates them in structured fashion - component-based UI with flow as well as document-based data visualizations. Both types of visual elements are implemented using the same basic patterns.
