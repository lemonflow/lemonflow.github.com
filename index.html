General

lemonflow is a framework for holisitic prototyping of animated data-driven user interfaces & visualizations. A specific strength of this framework is the support for predictive context-aware processing of information for dynamic visual presentation.

The framework provides as a reference implementation of certain novel concepts supporting this effort in a structured and validated approach. Exposing and applying such concepts in a easy and natural way can be very helpful when creating holistic prototypes of entire interactive experiences. Structure also can help to keep pace in projects, when rapidly changing visual ideas have to be prototyped quickly.

lemonflow enables the developer to bind data sources to drive visual properties of the elements on screen. It is thereby inspired and based on the work of Jeffrey Heer and his visualization frameworks prefuse, flare, protovis and D3. lemonflow implements concepts from these frameworks and extends the framework in several aspects:

  • merge and intersect componentized UIs with free-flow visualizations
  • adds states & transisitons, supports explicit animations (tweener-like)
  • supports dynamic data collections and data streams
  • build-in prediction/recommendation engine
  • context-aware information processing

Keywords: Animations, Meaningful Transitions, Visual Metaphors, Information/Data Visualization, UI components & flow, Information Architecture, Predictions, Granular Information System, Recommender, Personzalization, Adaptive UI, Context-aware Processing, Data Streams, Collaborative Filtering

Source Code Languages & Dependencies

Four implementations of lemonflow exist. The most complete one is the ActionsScript branch, requiring Flash/AIR for all graphics rendering. Javascript is the language, which the focus is on right now - we envision to reduce dependencies on proprietary modules

  • Java (deprecated):
    depends on the Java SWT GUI framework (Eclipse)
  • JavaScript-QT (experimental):
    runs with javascript runtime provided as integrated engine for QML Runtime; requires QT version minimum 5.0.1
  • ActionScript:
    compiles for and runs with adobe flash runtime, AVM2. Runs in Adobe AIR (tested versions up to AIR 3.3) and Flash Player (tested up to 11.3); lemonflow can use Stage3D, Flex or Standard Flash DisplayObjects as Render Surfaces. Stage3D is GPU-accelerated
  • JavaScript:
    ECMAScript 5; runs with a javascript runtime, e.g. Google V8; renders onto WebGL surface, HTML5 Canvas or divs. The JS source files in this repo are automatically generated from TypeScript; TypeScript is a superset of JavaScript which roughly goes in line with ECMAScript 6

Visual Operators (Custom Layouts)


package lemonflow.components.menubar
{
  import lemonflow.ui.base.InteractiveOperator;
  import lemonflow.ui.base.VisualItem;
  
  public class OperatorBarLayout extends lemonflow.ui.base.InteractiveOperator
  {
    protected var _view:* = null;
    private var floatingIdx:Number = 0;
    
    public var itemSpacing:int = 87;
    public var totalWidth:int = 940;
    public var offsetLeft:int = 40;
    public var offsetRight:int = 40;
    
    public var itemsWidths:Array = new Array();
    public var itemsWidthSum:Number = 0;
    public var itemsPositions:Array = new Array();
    
    public function OperatorBarLayout(v:*) {
      _view = v;
    }
    
    override public function update(slots:Vector. = null, currentIdx:Number = 0, nextIndex:Number = 0):void {
      itemsWidthSum = 0;
      for(var i:int=0; i<_view.slots.length; i++) {
        itemsWidths[i] = MenuBarItemRenderer(_view.renderers[i]).textMask.width;
        itemsWidthSum+=itemsWidths[i];
      }
      
      itemSpacing = _view.dataProvider.length>0?((totalWidth-itemsWidthSum)-(offsetLeft + offsetRight))/(_view.dataProvider.length-1):0;
      
      for(var k:int=0; k<_view.slots.length; k++) {
        itemsPositions[k] = (k>0?(itemsWidths[k-1]+itemSpacing):0);
        if(k>0) itemsPositions[k] += itemsPositions[k-1];
      }
      
      for(var j:int=0; j<_view.slots.length; j++) {
        floatingIdx = j-_view.currentIndex;
        _view.slots[j].x =  offsetLeft+itemsPositions[j];
        _view.slots[j].y =  _view.centerXYZ.y + 4;
        _view.slots[j].visible = (floatingIdx<=5);
      }

    }
  }
}

UI Components vs. Visualizations

One aspect which lemonflow addreses is the common (quite artifical) distinction in software architecture for component-based user interface frameworks (e.g. lists, menu bars, screens, tabs, side bars arranged in views) on one hand and then on the other hand frameworks for dynamic data visualizations (e.g. load data and apply a certain tree layout, encode values onto y-axis, force-based layout of nodes, draw pie chart based on data).

It might not be immediatly occurant why the underlying mechanisms in these frameworks are similar - but deep down there are common patterns and lemonflow merges these approaches. This provides an advantage for the developer/designer who does not need to choose one or the other framework for specific use cases

In many concrete modern user interfaces - the distinction between both types blurs, especially in the concept phase. Designers create information visualization screens as an interactive components - sometimes even as a visual metaphor for the entire UI (e.g. gravitation model of items). -- insert picture --

lemonflow suits both needs and integrates them in structured fashion.
